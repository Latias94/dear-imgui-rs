# DMV-050 - Multi-Window Input And Platform Feedback

Date: 2026-05-23

## Summary

Completed the input and platform-feedback slice for Bevy docking multi-viewport support.

The Bevy backend now routes input from the primary window and mapped Dear ImGui secondary viewport
windows into the same Dear ImGui IO stream. Secondary viewport mouse and touch events report the
hovered Dear ImGui viewport id and set `BackendFlags::HAS_MOUSE_HOVERED_VIEWPORT`.

Platform output is also viewport-aware: cursor and IME feedback are applied to the window named by
`PlatformImeData.ViewportId`, falling back to the primary window only when no mapped secondary
viewport is found. The viewport bridge now caches ECS-owned Bevy window position, size, focus, and
DPI state and exposes it through Dear ImGui PlatformIO getter callbacks.

## Decisions

- Keep PlatformIO getter callbacks world-free: Bevy systems synchronize window feedback into
  `ImguiViewportBridge`, and callbacks only read copied state.
- Set `ImguiBackendStatus::viewport_input_feedback_enabled` for native requested
  `multi-viewport` builds.
- Keep `ConfigFlags::VIEWPORTS_ENABLE` and `multi_viewport_supported` disabled until DMV-060 proves
  secondary viewport render routing.
- Bevy `0.19.0-rc.2` exposes a minimize request but not current minimized state, so the PlatformIO
  minimized getter returns `false` and the limitation is documented.

## Next
