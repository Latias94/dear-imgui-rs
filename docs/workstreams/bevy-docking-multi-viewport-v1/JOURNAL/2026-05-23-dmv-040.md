# DMV-040 - PlatformIO Enablement And Status

Date: 2026-05-23
Status: DONE

## Summary

Completed the status and platform gating slice for Bevy docking multi-viewport support.

The Bevy backend now reports multi-viewport capability as separate observable pieces:
request state, Cargo feature state, native target state, lifecycle bridge state, input feedback
state, render routing state, and full support. This keeps `multi_viewport_supported` false until
the remaining input and render contracts are proven, while still exposing that the native lifecycle
bridge is installed when requested.

The lifecycle bridge now installs only when `ImguiBackendConfig::multi_viewport` is true. Default
plugin configuration leaves `BackendPlatformUserData` unset and does not install the Bevy viewport
bridge.

## Decisions

- Treat native lifecycle support as a sub-capability, not as full Dear ImGui OS-level viewport
  support.
- Keep `ConfigFlags::VIEWPORTS_ENABLE` disabled until DMV-050 input feedback and DMV-060 secondary
  render routing are both implemented and verified.
- Keep wasm fail-closed. Normal wasm `render` builds compile; wasm with `multi-viewport` remains
  blocked by the core compile-time unsupported-target gate.

## Verification

- `cargo +stable nextest run -p dear-imgui-bevy multi_viewport --features render` - PASS.
- `cargo +stable nextest run -p dear-imgui-bevy viewport --features render,multi-viewport` - PASS.
- `cargo +stable check -p dear-imgui-bevy --target wasm32-unknown-unknown --features render` -
  PASS.
- `cargo +stable check -p dear-imgui-bevy --target wasm32-unknown-unknown --features render,multi-viewport`
  - EXPECTED FAIL at `dear-imgui/src/platform_io.rs`, which documents that wasm multi-viewport is
  unsupported.

Native sys crate builds still emit the existing macOS `ar -D` and C++ unused-function warnings.
