# DMV-020/030 - Core Contract And Bevy Lifecycle Bridge

Date: 2026-05-23
Status: DONE

## Summary

Completed the core contract audit and the first Bevy lifecycle bridge slice.

Core `dear-imgui-rs` did not need a new command ABI for this step. The existing typed PlatformIO
callback setters, `Io::BackendPlatformUserData`, and `Viewport` accessors are sufficient for an
engine backend to copy viewport state and enqueue intent without mutating engine state inside C ABI
callbacks.

The Bevy backend now has a `viewport` module behind the new `multi-viewport` feature. It installs
PlatformIO lifecycle callbacks, stores `ImguiViewportBridge` in `BackendPlatformUserData`, queues
viewport commands, and drains them from Bevy ECS to spawn, update, show, focus, retitle, and despawn
secondary `Window` entities.

## Decisions

- Keep the command queue backend-local for now; this is Bevy lifecycle state, not a generic core
  abstraction yet.
- Copy viewport state inside callbacks. Do not store raw `ImGuiViewport*` pointers beyond the call.
- Keep `ConfigFlags::VIEWPORTS_ENABLE` and `multi_viewport_supported` disabled. Lifecycle bridging
  alone is not full multi-viewport support because input feedback and secondary rendering are still
  missing.
- Clear `BackendPlatformUserData` and PlatformIO handlers when `ImguiContext` drops so Dear ImGui
  shutdown assertions remain satisfied.

## Verification

- `cargo +stable test -p dear-imgui-rs --features multi-viewport platform_io` - PASS.
- `cargo +stable test -p dear-imgui-bevy viewport --features render,multi-viewport` - PASS.

Both commands emitted existing native sys crate warnings from macOS `ar -D`/C++ unused functions;
there were no Rust test failures.
