# BBP-030 Docking Multi-Viewport Boundary Slice

Date: 2026-05-23

## Summary

Audited Dear ImGui multi-viewport support against the Bevy backend. The safe `dear-imgui` layer
already exposes `ConfigFlags::VIEWPORTS_ENABLE` plus PlatformIO platform and renderer callback
setters behind the `multi-viewport` feature. The missing piece is Bevy-specific: OS window
creation, destruction, focus, size, DPI, and per-window rendering must be bridged from C ABI
PlatformIO callbacks into Bevy ECS/window systems.

This slice therefore makes the current contract explicit instead of pretending the field is
implemented:

- `ImguiBackendStatus::multi_viewport_requested` records whether the user requested OS-level
  Dear ImGui platform windows.
- `ImguiBackendStatus::multi_viewport_supported` remains `false`.
- The lifecycle test proves the backend does not set `ConfigFlags::VIEWPORTS_ENABLE` before the
  Bevy PlatformIO bridge exists.
- README and crate docs now distinguish Bevy multi-window camera routing from Dear ImGui
  docking multi-viewport OS windows.

## Verification

- `cargo +stable nextest run -p dear-imgui-bevy plugin lifecycle --features render` - PASS, 5 tests
  run and 26 skipped by filter.

## Concerns

This is DONE_WITH_CONCERNS, not full OS-window support. The implementation should split into:

- a PlatformIO command queue that can safely cross from C ABI callbacks into Bevy systems;
- a viewport-to-window entity map and lifecycle cleanup policy;
- secondary platform-window render routing and focus/input feedback;
- full render-feature package and example gates after behavior changes.
